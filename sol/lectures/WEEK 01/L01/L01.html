1. What is an App?

Definition

An app (short for "application") is computer software or a program, most commonly a small, specific one used for mobile devices. Originally the term referred to any mobile or desktop application. With the rise of app stores (Apple's App Store, Google Play), the term evolved to refer to small programs that can be downloaded and installed all at once. In this course, we use "app" and "application" interchangeably, with a focus on web-based apps.

Explanation

Motivation: Apps solve a specific problem for a user. They range from email clients and word processors to social media platforms and code editors.

Key characteristic: An app is typically a focused, self-contained piece of software designed to perform a particular task or set of tasks.

Examples: Web browsers (Chrome, Firefox), email clients (Gmail, Outlook), word processors (MS Word), social media (Twitter, Instagram), code editors (VS Code), and even terminal emulators.

Key Definitions

  • Standalone app: An application that can operate without a network connection (e.g., a local text editor).
  • Network-oriented app: An application that depends on a network connection to function (e.g., Twitter, Instagram, web-based email).
  • Embedded application: A single-function, limited-scope program running on a dedicated device (e.g., digital camera firmware, microwave oven controller, smartwatch).

Common Mistakes & Misconceptions

  • Misconception: "An app is only for mobile phones."
    Why it's wrong: The term "app" now covers desktop, web, mobile, and embedded software.
    Correction: "App" is a generic term for application software regardless of platform.
    How to avoid: Think of "app" as shorthand for "application program."
  • Misconception: "An embedded application is always network-oriented."
    Why it's wrong: Many embedded apps (e.g., in a digital watch or calculator) have no network connectivity at all.
    Correction: Embedded apps are defined by their single-purpose nature, not by network connectivity.

Key Takeaways

* An app is a computer program designed to perform a specific task.
* Apps can be classified by their target platform: desktop, mobile, web, or embedded.
* In this course, the primary focus is on web-based apps that use a browser as the client interface.

2. Platforms: Desktop, Mobile, Web, and Embedded

Definition

A platform is the underlying hardware and operating system environment on which an app runs. Different platforms impose different constraints and offer different capabilities.

Explanation

Desktop: Keyboard, mouse, large video display. Uses the "desktop paradigm" (folders, files, documents). Often standalone; may have local data storage and optional network connection. SDKs are usually OS-specific (Windows, macOS, Linux).

Mobile: Targeted at phones and tablets. Constraints: limited screen space, touch/audio/camera interfaces, limited memory and processing power, battery-powered. Frameworks may be OS-specific (Swift for iOS) or cross-platform (Flutter, React Native). Usually strongly network-oriented.

Web: The platform of choice for this course. Works across OS and device types. Heavily network-dependent (though offline workarounds exist). Built on HTTP, HTML, CSS, and JavaScript. Provides a common base that abstracts away hardware differences.

Embedded: Single-function, limited-scope devices (digital cameras, smart refrigerators, microwaves). Highly specialized; often have minimal user interfaces.

Comparison with Related Concepts

Platform Primary Interface Network Dependency Key Constraint
Desktop Keyboard, mouse, large screen Optional OS-specific SDKs
Mobile Touch, voice, camera, sensors Usually high Battery, memory, screen size
Web Browser (cross-platform) High (offline via service workers) Browser compatibility
Embedded Minimal / task-specific Low or none Single purpose, limited scope

Key Takeaways

* The web is a powerful cross-platform environment because browsers behave consistently across OSes.
* Web apps can leverage cloud storage for persistence and "access anywhere" capability.
* Embedded apps are characterized by being dedicated to a single function.

3. Components of an App: Storage, Computation, Presentation

Definition

Every app, regardless of platform, can be decomposed into three fundamental components: Storage (where data lives), Computation (how data is manipulated), and Presentation (how data is shown to the user).

Explanation

Storage: The component where application data is kept, either permanently (databases, cloud servers, local files) or temporarily (caches). In an email app, storage handles where emails reside — on the server (Gmail) or locally (Outlook PST files). Storage formats include proprietary databases (Google's internal systems), mbox/maildir (Thunderbird), or relational databases.

Computation: The "business logic" — indexing emails, searching, sorting, deleting, archiving. This is the controller's domain in MVC. Computation manipulates data stored in the storage component based on user requests.

Presentation: The user-facing side — displaying a list of emails, rendering individual emails, formatting text, showing unread counts. This is the view's responsibility. On the web, presentation is handled by HTML, CSS, and client-side JavaScript.

Intuition / Mental Model

The Restaurant Analogy: Storage is the pantry (ingredients stored), Computation is the chef (prepares and manipulates ingredients), Presentation is the waiter (brings the finished dish to the customer in an appealing way).

Key Takeaways

* Storage can be permanent or temporary; it's where data resides.
* Computation manipulates data — this is the "business logic."
* Presentation is the interface through which the user sees and interacts with data.

4. Client-Server Architecture

Definition

The client-server model is an architecture where there is an explicit differentiation between servers (which store data and provide it on demand) and clients (end users that request data and handle user interaction). A network connects them, acting as a data pipe.

Explanation

Server: Stores data, provides data on demand, may perform computations. Can be a single machine or a cluster behind a load balancer.

Client: End-user device (laptop, phone). Requests data, handles display and user interaction. May be a machine client (e.g., antivirus updater) with no direct user interaction.

Network: Connects client to server. Transmits data without altering it. Can be local (both on same machine) or remote.

Variants: Multiple servers, single queue, multiple queues, load-balancing frontends. Examples: Email (Gmail), Databases, WhatsApp/messaging, Web browsing.

Key Takeaways

* Clear separation: server stores and computes; client requests and displays.
* Data can flow both ways — clients can also send data to servers (e.g., sending an email).
* The same machine can host both client and server software, still conceptually a client-server system.

5. Peer-to-Peer (Distributed) Architecture

Definition

In a peer-to-peer (P2P) model, all nodes ("peers") are considered equivalent. There is no fixed distinction between client and server — any node can both request and provide data. Some peers may be "more equal" (e.g., high-bandwidth servers acting as introducers).

Explanation

Key features: Error tolerance — if a central server crashes, peers can re-elect a new master. Shared information across nodes. Data may be stored across multiple devices in encrypted form.

Examples: BitTorrent (file sharing where downloaders also upload), Blockchain-based systems (distributed trust), IPFS and Tahoe (distributed file systems).

Limitations: Not suitable for all applications. Email, for instance, is not a good P2P candidate because you don't want your emails stored on strangers' machines. P2P works best when data can be shared publicly or encrypted.

Key Takeaways

* P2P eliminates the single point of failure but adds complexity in coordination.
* All peers can serve as both client and server simultaneously.
* Not all apps are suited for P2P — trust and data privacy are major considerations.

6. Software Design Patterns & MVC

Definition

A design pattern is "a general, reusable solution to a commonly occurring problem within a given context in software design." Experienced designers observe patterns in code; reusing them makes design and development faster and guides the thought process.

MVC (Model-View-Controller) is a design pattern that separates an application into three interconnected components:

  • Model: Core data and business rules. Manages data, logic, and rules of the application. Responsible for adding, deleting, updating, and fetching data from the database.
  • View: User-facing side. Displays data and provides interfaces for interaction. A "view" does not always refer to what the user can see — it could be an audio interface or API response.
  • Controller: "Business logic" — how to manipulate data. Executes the business logic of the application. Can interact with both the model and the view.

Explanation

Origins: Smalltalk language, 1979. Despite its age, MVC remains a foundational concept in modern web frameworks (Ruby on Rails, Django, ASP.NET MVC, Laravel).

Data flow: User → Controller → Model → View → User. The view cannot directly manipulate the model; all changes go through the controller. The developer can create multiple views of the same model.

Alternatives: Model-View-Adapter (MVA), Model-View-Presenter (MVP), Model-View-ViewModel (MVVM), Hierarchical MVC (HMVC). Each has its uses, but fundamentals are very similar.

Common Mistakes & Misconceptions

  • Misconception: "MVC is the only architecture to build a web app."
    Why it's wrong: Many alternatives exist (MVP, MVVM, etc.). MVC is one of many valid patterns.
    Correction: MVC is a widely used and well-understood pattern, but not the only one.
  • Misconception: "The model controls user interaction."
    Why it's wrong: The controller handles user interaction and business logic; the model manages data.
    Correction: Model = data; View = display; Controller = logic and interaction.

Key Takeaways

* MVC separates concerns: data (Model), display (View), and logic (Controller).
* Business logic belongs in the Controller, not the Model or View.
* Design patterns are guidelines, not rigid rules — they guide thinking, not constrain it.

7. Historical Background: From Telephones to the Internet

Definition

The modern internet evolved through several key stages: circuit-switched telephone networks (~1890s), packet-switched networks (~1960s), ARPANET (1969), the Internet Protocol (IP, 1983), Transmission Control Protocol (TCP, 1983), Domain Name System (~1985), and HyperText / World Wide Web (~1989+).

Explanation

Circuit Switching (1890s+): In telephone networks, a dedicated physical path (closed loop) was established between caller and receiver. Physical wires were tied up for the entire call duration, even during silence — leading to resource underutilization. Communication happened in real time.

Packet Switching (~1960s): Data is broken into packets; wires are occupied only when data is being sent. A single set of wires can carry information between two exchanges for multiple conversations. More efficient use of infrastructure. Data (not just voice) can be transmitted.

ARPANET (1969): Advanced Research Projects Agency Network. A node-to-node network, mostly university-driven. Other networks existed (IBM SNA, DEC DECNet, Xerox Ethernet) but each had its own protocol, making inter-network communication difficult.

Internet Protocol (IP, Jan 1, 1983): Defined standardized headers, packet types, and interpretation. Can be carried over different underlying networks (Ethernet, DECnet, PPP, SLIP). It bridges different network protocols.

TCP (1983): Transmission Control Protocol establishes reliable communication with acknowledgments, retry, and error control. Automatically scales and adjusts to network limits. TCP is a session-establishment protocol between client and server.

Domain Names (~1985): Replaced numeric IP addresses with human-readable names (e.g., apple.com). Hierarchical structure with root servers and sub-domain servers. Allowed IP addresses to change transparently behind the scenes.

HyperText (~1989+): Tim Berners-Lee at CERN created the concept of text documents with embedded formatting and links to other documents. This became the World Wide Web.

Comparison: Circuit Switching vs. Packet Switching

Aspect Circuit Switching Packet Switching
Path Dedicated physical path (closed loop) Shared wires; packets take any available path
Resource usage Wires tied up for entire call duration Wires occupied only when data is sent
Efficiency Lower (idle time wastes resources) Higher (statistical multiplexing)
Data type Primarily voice Voice, data, video — anything digitized
Example 1890s telephone networks Modern internet

Key Takeaways

* ARPANET stands for Advanced Research Projects Agency Network.
* A protocol is a set of rules that defines how data packets are formed and placed on wires.
* IP bridges different network protocols and defines standardized headers.
* TCP provides reliability through acknowledgments and retransmission; UDP does not.
* The Internet is a network of networks; the World Wide Web is a system of interconnected resources over the internet.

8. TCP vs. UDP

Definition

TCP (Transmission Control Protocol) is a connection-oriented protocol that prioritizes reliability of data delivery through acknowledgments, retransmission, and error control. UDP (User Datagram Protocol) is a connectionless protocol that sends data without guarantees — it can result in loss of data but is faster and has lower overhead.

Comparison

Aspect TCP UDP
Connection Connection-oriented Connectionless
Reliability High (acknowledgments, retry) Low (no guarantee of delivery)
Acknowledgment Yes No
Data loss Minimized via retransmission Can result in loss of data
Use case Web browsing, email, file transfer Streaming, gaming, VoIP

Key Takeaways

* TCP gives priority to reliability; UDP gives priority to speed.
* Only TCP provides acknowledgment after receiving data.
* UDP can result in loss of data — suitable for real-time applications where occasional loss is acceptable.

9. World Wide Web & Web 2.0

Definition

The World Wide Web is a system of interconnected hypertext documents (resources) accessed via the internet. Web 2.0 (~2004+) refers to the evolution where HTTP became a general transport mechanism, pages became dynamically generated, and client-side computation (JavaScript) enabled rich interactivity.

Explanation

Original Web limitations: Static pages, complicated executable interfaces (CGI was the only way), limited styling, and browser compatibility issues ("browser wars" of the 1990s).

Web 2.0 characteristics: Dynamic pages generated on the fly; HTTP as a transport mechanism for any data (binary, serialized objects); client-side computation and rendering via JavaScript; platform-agnostic — the browser becomes almost an operating system. This enabled shopping carts, user tracking, personalized content, and rich web applications.

Key Takeaways

* The World Wide Web is NOT the same as the Internet — the Web is a system of resources over the Internet.
* The Internet is a network of networks; the Web is a system of interconnected resources over the internet.
* Web 2.0 transformed the web from a document delivery system into an application platform.

10. Web Servers: Definition and Basic Operation

Definition

A web server is any computer (or virtual machine) with a network connection that runs software to listen for incoming network connections on a fixed port and respond to client requests in specific ways. The term "web server" can refer to the software (e.g., Apache, Nginx), the hardware machine, or both.

Explanation

Core functions of a web server:

  1. Listen on a fixed port (commonly port 80 for HTTP, 443 for HTTPS, or custom ports like 8000, 8080).
  2. On incoming request, run some code and return a result.
  3. Send standard headers as part of the result (status code, content type, content length, etc.).
  4. Output can be text or other format — specified via MIME types.

What a web server does NOT do: It does NOT maintain the state of the client (HTTP is stateless). The operating system handles the low-level details of ports and network connections; the web server works one level above.

Examples of web servers: Apache Tomcat Server, IIS (Internet Information Server), Mongoose, Python's built-in HTTP server. Python is NOT a web server — it's a programming language that can be used to run a web server.

Key Takeaways

* Any computer with a network connection can act as a web server.
* A web server accepts incoming connections on a fixed port and responds in specific ways.
* The computer running the web server program is also called a "web server."
* Python is not a web server; it is a programming language that can run web server software.

11. HTTP Protocol: Requests, Responses, Headers, and Status Codes

Definition

HTTP (HyperText Transfer Protocol) is a text-based protocol that defines how clients and servers communicate. It specifies the format of requests (GET, POST, etc.) and responses (status codes, headers, data). HTTP is an abbreviation for Hypertext Transfer Protocol — it is a text-based protocol (not image-based).

Explanation

Typical HTTP Request:

GET / HTTP/1.1
Host: localhost:1500
User-Agent: curl/7.64.1
Accept: */*

Components: GET is the method; / is the path; HTTP/1.1 is the protocol version. Host specifies which host to connect to (required in HTTP/1.1 for virtual hosting). User-Agent identifies the client. Accept specifies what MIME types the client can handle — */* means "accept any form of data." A blank line signals the end of headers.

Typical HTTP Response:

HTTP/1.1 200 OK
Server: SimpleHTTP/0.6 Python/3.9.2
Date: Thu, 17 Jun 2021 03:11:15 GMT
Content-type: text/html
Content-Length: 31
Last-Modified: Thu, 17 Jun 2021 03:10:18 GMT

<h1>Hello world</h1>

Status Codes: 200 OK (success), 404 Not Found (resource missing), 500 Internal Server Error (server crashed). 301 indicates a page has moved permanently.

GET vs. POST:

  • GET: Generally used to request a resource from the server. Data is visible in the URL (query string). Used for simple requests, search queries.
  • POST: Generally used to submit forms. Data is NOT visible in the URL. Used for complex form data, large text blocks, file uploads.

Both GET and POST are methods used by HTTP to transfer data between a browser and a server.

Key Definitions

  • MIME: Multipurpose Internet Mail Extensions — originally for email attachments, now used to specify content types (text/html, application/pdf, image/png, etc.). The expanded form is NOT "Multimedia Intranet Mail Extension" — that's a common distractor.
  • CGI: Common Gateway Interface — an interface specification that enables web servers to execute an application program. It is an abbreviation for "Common Gateway Interface."
  • Accept: image/* means the client is able to understand images only (any image format).
  • Accept: */* means the client is willing to accept any form of data.

Key Takeaways

* HTTP is a text-based protocol, not image-based.
* Both GET and POST are valid HTTP methods for transferring data.
* Data is visible in the URL with GET; hidden in the body with POST.
* A protocol is a set of rules agreed to by both client and server in a communication.
* FTP stands for File Transfer Protocol (NOT "File Traffic Protocol").
* HTTP is a stateless protocol (it does NOT maintain client state between requests).

12. Building a Simple Web Server

Definition

A minimal web server can be written in a single line of shell script. The simplest possible server listens on a port, and whenever a request arrives, it responds with a fixed HTTP header and some dynamic content (like the current date).

Explanation

The one-line web server:

while true; do echo -e "HTTP/1.1 200 OK\n\n $(date)" | nc -l localhost 1500; done

Breakdown:

  • while true; do ... done — infinite loop; keeps the server running.
  • echo -e "HTTP/1.1 200 OK\n\n $(date)" — prints the HTTP response header (200 OK), two newlines (separating headers from body), and the current date as the body.
  • | (pipe) — redirects the output of echo into the next command.
  • nc -l localhost 1500 — Netcat listens on localhost port 1500; when a connection comes in, it sends the piped data and exits, then the while loop restarts it.

Testing with curl:

curl -v http://localhost:1500

The -v (verbose) flag shows the full request and response headers, which is very useful for debugging.

Python's built-in HTTP server:

python -m http.server

This serves files from the current directory on port 8000 by default. It understands basic HTTP requests and provides proper headers including Content-type, Content-Length, and Last-Modified. By convention, if a directory is requested, it looks for an index.html file to serve.

Key Definitions

  • Loopback address: 127.0.0.1 (IPv4) and ::1 (IPv6) — addresses that refer to the local machine itself. The addresses 128.0.0.8 and 1::.1 are NOT valid loopback addresses.
  • 0.0.0.0 in IPv6: Represented as :: or ::0 (all zeros). Not 10::1 or 01::0.
  • URL components: In http://myserver.com/index.html?uid=10&sid=4032013, the protocol is http, the host-name is myserver.com, the file-path is index.html, and the query-string is uid=10&sid=4032013.

Common Mistakes & Misconceptions

  • Misconception: "onlinedegree.iitm.ac.in/academics.html is a domain name."
    Why it's wrong: A domain name is just the host part (e.g., onlinedegree.iitm.ac.in). The full string including the path is a URL.
    Correction: The full string https://onlinedegree.iitm.ac.in/academics.html is a URL. HTTPS is the protocol.

Key Takeaways

* A web server can be as simple as a one-line shell script.
* curl is invaluable for debugging HTTP requests and responses.
* The blank line between headers and body is critical — it signals the end of headers.
* Valid IPv4 addresses have four octets, each between 0 and 255 (e.g., 192.168.64.34 — valid; 192.168.256.1 — invalid).

13. URL Anatomy and Components

Definition

A URL (Uniform Resource Locator) is a reference to a web resource that specifies its location and the protocol used to access it. Key components: protocol, host-name (or host-IP), port, file-path, and query-string.

Explanation

Example: http://myserver.com/index.html?uid=10&sid=4032013

  • Protocol: http
  • Host-name: myserver.com
  • File-path: /index.html
  • Query-string: uid=10&sid=4032013

In curl http://127.0.0.1:8080: Protocol is http, host-IP is 127.0.0.1, port is 8080, and the command is curl.

Key Takeaways

* The protocol, host, port, path, and query string are all distinct parts of a URL.
* When you send a request to https://www.google.com/, from the server's point of view, the Web Browser is the client — not the Google home page or the Google server.

14. Web Browser Characteristics

Definition

A web browser is an application program used to display web documents. It sends HTTP requests to servers and receives HTTP responses, then renders the content for the user.

Explanation

True statements about web browsers:

  • A web browser is an application program to display web documents. ✓
  • A web browser sends an HTTP request and receives an HTTP response. ✓

False statements:

  • A web browser is installed in a web server machine and hosts websites. ✗ (That's a web server's job.)
  • A web browser is a web app. ✗ (A browser is a desktop/mobile application that renders web apps.)

Key Takeaways

* The browser is the client in the client-server model for web communication.
* Browsers interpret HTML, CSS, and JavaScript to render web pages.

15. Performance of a Website: Latency, Bandwidth, Memory, and Storage

Definition

Website performance is determined by several factors: latency (round-trip time for requests), bandwidth (data transfer rate), server memory (RAM), and storage capacity. These constraints dictate how many concurrent users a server can handle and how quickly it can respond.

Latency

RTT (Round Trip Time) is the time taken for a request to travel from client to server and back. Speed of light in fiber cable ≈ \(2 \times 10^8\) m/s ≈ 5 ns/m. For a distance of 1,000 km: ~5 ms one way. For 2,000 km: ~10 ms one way, ~20 ms round trip. This limits the client to a maximum of ~50 requests per second (if each request must complete before the next begins).

\[ \text{RTT} = \frac{2 \times \text{distance}}{\text{speed of light in cable}} = \frac{2 \times 12,\!500,\!000\ \text{m}}{2 \times 10^8\ \text{m/s}} = 0.125\ \text{s} = 125\ \text{ms} \]

As distance increases, the maximum number of requests per second decreases.

Bandwidth

Response size × requests/second = required bandwidth. Example: 10,000 requests/sec × 150 kB = 1,500,000 kB/s = 1.5 GB/s ≈ 12 Gbps. Bandwidth needed = \( \text{requests/sec} \times \text{response size} \times 8 \) (to convert bytes to bits).

\[ \text{Bandwidth} = 6,\!250 \times 200\ \text{kB} \times 8 = 6,\!250 \times 1,\!600\ \text{kb} = 10,\!000,\!000\ \text{kb/s} = 10\ \text{Gbps} \]

Memory

Each server process consumes RAM. One Python HTTP server process ≈ 6 MB. For 2 million concurrent viewers (e.g., YouTube live stream): \( 2,\!000,\!000 \times 6\ \text{MB} = 12,\!000,\!000\ \text{MB} \approx 12\ \text{TB} \) of RAM. For 5 million concurrent viewers of a 2 MB video: \( 5,\!000,\!000 \times 2\ \text{MB} = 10,\!000,\!000\ \text{MB} = 10\ \text{TB} \) of RAM minimum. This is why large-scale services distribute load across thousands of servers.

Storage

Google indexes hundreds of billions of web pages. Estimated total index size: ~100 Petabytes (100,000,000 GB). This requires massive distributed storage systems with efficient retrieval mechanisms.

Key Takeaways

* RTT stands for Round Trip Time, not "Round Tip Time" or "Robust Travel Time."
* Latency is fundamentally limited by the speed of light — even with instant server processing, distance creates unavoidable delay.
* As distance increases, the max requests per second decreases.
* Scaling requires distributing load across many servers, not just building bigger single machines.
* The correct order of tasks when requesting a URL: (4) DNS lookup → (1) HTTP request → (3) server response → (2) browser assembles and displays.
