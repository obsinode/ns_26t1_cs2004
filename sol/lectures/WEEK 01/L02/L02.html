1. What is an App? – Definition, Scope, and Platform Classification

Definition

An app (short for "application") is a computer program designed to perform a specific task or set of tasks. According to the Techopedia definition cited in the lecture: "An app is computer software, or a program, most commonly a small, specific one used for mobile devices. The term app originally referred to any mobile or desktop application, but as more app stores have emerged… the term has evolved to refer to small programs that can be downloaded and installed all at once."

Explanation

Motivation: Apps solve focused problems for users. They range from email clients and word processors to social media platforms and code editors. The term "app" is now used interchangeably with "application" across all platforms — desktop, mobile, web, and embedded.

Historical context: Originally, the term "app" was closely associated with mobile apps after the Apple iPhone popularized the App Store concept. Over time, even desktop software and web-based tools began being called "apps."

Key characteristic: An app is typically a focused, self-contained piece of software designed to perform a particular task or set of tasks — not a sprawling general-purpose system.

Key distinction: An electronic calculator is an example of an embedded application. An embedded application is only used to perform a specific task. An embedded application can be installed in a device — it is not restricted from being installed. An embedded application is not always network-oriented.

Platform Classification

Platform Primary Interface Network Dependency Key Constraints Examples
Desktop Keyboard, mouse, large screen Optional (offline-capable) OS-specific SDKs MS Word, VS Code, Firefox
Mobile Touch, voice, camera, sensors Usually high Battery, memory, screen size Instagram, WhatsApp, Amazon
Web Browser (cross-platform) High (offline via service workers) Browser compatibility Gmail (web), Google Docs
Embedded Minimal / task-specific Low or none Single purpose, limited scope Digital camera, microwave, smartwatch
Misconception: "An app is only for mobile phones."
Why it's wrong: The term "app" now covers desktop, web, mobile, and embedded software.
Correction: "App" is a generic term for application software regardless of platform.
How to avoid: Think of "app" as shorthand for "application program."

Key Takeaways

* An app is a computer program designed to perform a specific task.
* Apps can be classified by their target platform: desktop, mobile, web, or embedded.
* In this course, the primary focus is on web-based apps that use a browser as the client interface.
* Google, Facebook, and PowerShell are not examples of embedded applications — an electronic calculator is.

2. Components of an App – Storage, Computation, Presentation

Definition

Every app can be decomposed into three fundamental components: Storage (where data lives), Computation (how data is manipulated), and Presentation (how data is shown to the user). This is the foundational abstraction that underlies the MVC pattern.

Detailed Explanation

Storage: The component where application data is kept — either permanently (databases, cloud servers, local files) or temporarily (caches). Storage is a component of an application where we can store our application data permanently and where data can be stored temporarily. Storage does not manipulate data based on requests — that's Computation's job. Storage is not an interface for rendering data — that's Presentation's job.

Computation: The "business logic" — indexing emails, searching, sorting, deleting, archiving, filtering. This is the controller's domain in MVC. Computation manipulates data stored in the storage component based on user requests. In the Gmail app, filtering emails to display only those received in March is a Computation task.

Presentation: The user-facing side — displaying a list of emails, rendering individual emails, formatting text, showing unread counts. This is the view's responsibility. In an email client, the component used to display the list of emails is Presentation.

Intuition / Mental Model

The Restaurant Analogy: Storage is the pantry (ingredients stored), Computation is the chef (prepares and manipulates ingredients), Presentation is the waiter (brings the finished dish to the customer in an appealing way).
Storage component truth: Storage is a component of an application where we can store our application data permanently AND where data can be stored temporarily. It is NOT used to manipulate data based on request (that's Computation). It is NOT an interface on which the user-requested data is rendered (that's Presentation).

Key Takeaways

* Storage can be permanent or temporary; it's where data resides — NOT where it's manipulated or displayed.
* Computation manipulates data — filtering, indexing, searching — this is the "business logic."
* Presentation is the interface through which the user sees and interacts with data.
* These three components map directly to the MVC pattern: Storage → Model, Computation → Controller, Presentation → View.

3. Client-Server Architecture – The Foundational Network Model

Definition

The client-server model is an architecture where there is an explicit differentiation between servers (which store data and provide it on demand) and clients (end users that request data and handle user interaction). A network connects them, acting as a transparent data pipe.

Mechanism – Three Components

Server: Stores data, provides data on demand, may perform computations. Can be a single machine or a cluster behind a load balancer.

Client: End-user device (laptop, phone). Requests data, handles display and user interaction. May be a machine client (e.g., antivirus updater) with no direct user interaction.

Network: Connects client to server. Transmits data without altering it. Can be local (both on same machine) or remote. Acts solely as a data pipe — no modifications to the content.

Examples of Client-Server: Email (Gmail), Databases, WhatsApp/messaging, Web browsing. In each case, there is a well-defined server that stores and manages data, and clients that request and display information.

Variants

Multiple servers, single queue, multiple queues, load-balancing frontends. Even when both client and server run on the same machine (e.g., a local web server for development), the conceptual separation remains: one process listens (server), another connects (client).

Misconception: "Data can only flow from server to client."
Correction: Data flows both ways — clients send data (e.g., POST requests, form submissions), and servers respond with data. The network is bidirectional.

Key Takeaways

* Clear separation: server stores and computes; client requests and displays.
* The network is a transparent pipe — it does not modify the data in transit.
* The same machine can host both client and server software — still conceptually a client-server system.

4. Peer-to-Peer (P2P) Architecture – The Distributed Alternative

Definition

In a peer-to-peer (P2P) model, all nodes ("peers") are considered equivalent. There is no fixed distinction between client and server — any node can both request and provide data. Some peers may be "more equal" than others (e.g., high-bandwidth servers acting as introducers or masters).

Key Features

Error tolerance: If a central server crashes, peers can re-elect a new master. The network degrades gracefully rather than failing completely.

Shared information across nodes: Data may be stored across multiple devices in encrypted form — no single machine holds the complete dataset.

Examples: BitTorrent (file sharing where downloaders also upload), Blockchain-based systems (distributed trust via consensus), IPFS and Tahoe (distributed file systems).

Limitations

Not suitable for all applications. Email, for instance, is not a good P2P candidate because users don't want their emails stored on strangers' machines. P2P works best when data can be shared publicly or encrypted, and when trust is distributed across the network rather than centralized.

P2P vs. Client-Server: In client-server, the roles are fixed — the server always serves and the client always requests. In P2P, any peer can act as either at any time. P2P eliminates the single point of failure but adds coordination complexity.

Key Takeaways

* P2P eliminates the single point of failure but adds complexity in coordination.
* All peers can serve as both client and server simultaneously.
* Not all apps are suited for P2P — trust and data privacy are major considerations.

5. Software Design Patterns & The MVC Architecture

Definition of a Design Pattern

A design pattern is "a general, reusable solution to a commonly occurring problem within a given context in software design." (Src: Wikipedia). Experienced designers observe patterns in code; reusing them makes design and development faster and guides the thought process.

MVC – Model • View • Controller

Origins: Smalltalk programming language, 1979. Despite its age, MVC remains a foundational concept in modern web frameworks (Ruby on Rails, Django, ASP.NET MVC, Laravel).

Component Responsibility What it does NOT do
Model Core data and business rules. Manages data, logic, and rules. Responsible for adding, deleting, updating, and fetching data from the database. Does NOT control user interaction. Does NOT handle display logic.
View User-facing side. Displays data and provides interfaces for interaction. Also receives data from the user. Does NOT directly modify the Model. Does NOT contain business logic.
Controller Executes the business logic of an application. Can interact with both Model and View. Handles user input interpretation. Does NOT store core data. Should NOT contain business rules — those belong in the Model.

Data Flow (Unidirectional Cycle)

User → Controller → Model → View → User

The view cannot manipulate the model directly — all changes go through the Controller. The developer can create multiple views of a single model. In MVC, the 'View' does not always refer to what the user can see — it could be an audio interface or an API response.

Where does business logic belong? In the Controller — the controller executes the business logic of an application. The Model manages data and rules (e.g., "counter cannot exceed 100"), but the Controller orchestrates when and how those rules are invoked.
Misconception 1: "MVC is the only architecture to build a web app."
Correction: Many alternatives exist (MVP, MVVM, HMVC). MVC is one of many valid patterns — it is NOT the only one.

Misconception 2: "Any modification in the user interface will always affect the model."
Correction: In MVC, the View and Model are decoupled — changes in the UI do NOT automatically affect the Model. The Controller mediates all changes.

Misconception 3: "The database can be directly updated by the end user/client."
Correction: In MVC, the Model encapsulates the database — the end user never directly updates it. All updates flow through Controller → Model.

Alternative Patterns

Model-View-Adapter (MVA), Model-View-Presenter (MVP), Model-View-ViewModel (MVVM), Hierarchical MVC (HMVC). Each has its uses, but the fundamentals are very similar — separation of data, display, and logic.

Key Takeaways

* MVC separates concerns: data (Model), display (View), and logic (Controller).
* Business logic is included in the Controller.
* The Model is responsible for adding, deleting, updating, and fetching data from the database.
* The View is responsible for displaying data AND receiving data from the user.
* Design patterns are guidelines, not rigid rules — they guide thinking, not constrain it.

6. Three-Tier Architecture – Presentation, Business, and Persistence Layers

Definition

In a web application, a three-tier architecture separates the system into three logical layers: the Presentation Layer (provides the interface to the end user), the Business Layer (executes the application's core logic and computations), and the Persistence Layer (manages data storage and retrieval).

Layer Responsibilities

Layer Responsibility Example Task
Presentation Layer Provides the interface to the end user Rendering the HTML page, handling button clicks
Business Layer Executes business logic and computations Calculating simple interest, filtering emails by date
Persistence Layer Manages data storage and retrieval Storing user profiles in a database
Integration Layer Connects to external systems and APIs Calling a payment gateway
Important distinction: The Presentation Layer provides the interface to the end user — it is NOT the Business Layer. If a web application can perform simple interest calculations, the Business Layer should perform that task — not the Presentation or Persistence layer.
* The Presentation Layer provides the interface to the end user.
* The Business Layer handles calculations and business logic (e.g., simple interest).
* The Persistence Layer manages data storage and retrieval.
* This maps closely to MVC: Presentation ≈ View, Business ≈ Controller, Persistence ≈ Model.

7. History of the Web – From Circuit Switching to the World Wide Web

The Evolution Timeline

Era Technology Key Characteristics
~1890s Circuit-Switched Telephone Networks Dedicated physical path (closed loop). Wires tied up for entire call duration — even during silence. Communication happens in real time. Creates a dedicated path — but resources are underutilized during silence.
~1960s Packet-Switched Networks Data broken into packets. Wires occupied only when data is sent. Single set of wires can carry multiple conversations. More efficient. Data (not just voice) can be transmitted.
1969 ARPANET Advanced Research Projects Agency Network. Node-to-node, mostly university-driven. IBM SNA, DEC DECNet, Xerox Ethernet also existed — each with its own protocol.
1983 TCP/IP Internet Protocol (IP) defines standardized headers and bridges different network protocols. Transmission Control Protocol (TCP) establishes reliable communication with acknowledgments and error control.
~1985 Domain Name System (DNS) Replaces numeric IP addresses with human-readable names. Hierarchical structure. Abstracts away from individual server machines.
~1989+ HyperText & World Wide Web Tim Berners-Lee at CERN. Text documents with embedded formatting and links to other documents. The Web is a system of interconnected resources over the internet.
~2004+ Web 2.0 Dynamic pages generated on the fly. HTTP as a transport mechanism for any data. Client-side computation and rendering via JavaScript. Platform-agnostic — the browser becomes almost an OS.
ARPANET stands for: Advanced Research Projects Agency Network — NOT "American Research Projects Agency Network" and NOT "Advanced Research Protocol Agency Network."

A Protocol – Formal Definition

A protocol is a set of rules that defines how data packets are formed and placed on wires. It is an agreement between both client and server (not just the client alone). HTTP is a protocol. A protocol describes formatting, transmission, and receiving of data between two network devices.

Misconception: "A protocol is rules agreed to by the client only."
Correction: A protocol requires agreement from both the client and server in a communication — both sides must follow the same rules for successful communication.

Key Takeaways

* The Internet is a network of networks — it is NOT a "system of interconnected resources" (that's the Web) and NOT "a set of rules" (that's a protocol).
* The World Wide Web is a system of interconnected resources over the internet — it is NOT "a network of networks" (that's the Internet).
* In circuit-switched networks, physical wires form a closed loop — the same set of wires cannot be used by anyone else while a connection is active (which is true about circuit-switched networks).
* In packet-switched networks, a single set of wires can carry information between two exchanges.
* The information in a packet-switched network is NOT carried when data is not being sent — wires are occupied only when data is transmitted.

8. TCP vs. UDP – Reliability vs. Speed

Definition

TCP (Transmission Control Protocol) is a connection-oriented protocol that prioritizes reliability of data delivery through acknowledgments, retransmission, and error control. UDP (User Datagram Protocol) is a connectionless protocol that sends data without guarantees — it can result in loss of data but is faster and has lower overhead.

Detailed Comparison

Aspect TCP UDP
Connection Connection-oriented Connectionless
Reliability High — acknowledgments, retry, error control Low — no guarantee of delivery; can result in loss of data
Acknowledgment Yes — provides acknowledgment after receiving data No — does not provide acknowledgment
Data loss Minimized via retransmission Can result in loss of data
Use case Web browsing, email, file transfer Streaming, gaming, VoIP
Reliability priority TCP gives priority to reliability of data delivered UDP gives priority to speed over reliability
Session establishment protocol: TCP/IP is a session establishment protocol between a client and a server — NOT ARP, DHCP, or UDP.
Misconception: "UDP is more reliable than TCP."
Correction: UDP is LESS reliable — it does not guarantee delivery. TCP is the reliable protocol.

Misconception: "Both UDP and TCP provide acknowledgment after receiving data."
Correction: Only TCP provides acknowledgment. UDP does not.

Key Takeaways

* TCP gives priority to reliability; UDP gives priority to speed.
* Only TCP provides acknowledgment after receiving data.
* UDP can result in loss of data — suitable for real-time applications where occasional loss is acceptable.
* TCP/IP is a session establishment protocol; UDP is not.

9. Internet Protocol (IP) – The Universal Network Layer

Definition

Internet Protocol (IP) is a set of rules that specifies one way to deliver data over the network. It defines standardized headers for all network protocols, bridges different network protocols (Ethernet, DECnet, PPP, SLIP), and can be carried over different underlying networks.

What IP Does

  • It bridges different network protocols — allows Ethernet-based networks to communicate with DECnet-based networks by adding a common IP header.
  • It defines a standardized header for all the network protocols — providing a common format that all networks can interpret.

What IP Does NOT Do

  • It does NOT establish reliable communication — that's TCP's job. IP is inherently unreliable (best-effort delivery).
  • It does NOT require a predefined specific network — it is designed to be carried over by its underlying network, whatever that may be.
Misconception: "IP establishes reliable communication."
Correction: IP is a best-effort protocol — it does NOT guarantee delivery. TCP (which runs on top of IP) provides reliability through acknowledgments and retransmission.

Key Takeaways

* IP bridges different network protocols and defines standardized headers.
* IP does NOT require a predefined specific network — it can be carried over various physical networks (Ethernet, Wi-Fi, fiber, etc.).
* IP does NOT establish reliable communication — that's TCP's responsibility.
* The protocol that gives priority to reliability of data delivered is TCP, not IP.

10. Web Servers – Definition, Functions, and Examples

Definition

A web server is any computer (or virtual machine) with a network connection that runs software to listen for incoming network connections on a fixed port and respond to client requests in specific ways. The term "web server" can refer to the software (e.g., Apache, Nginx), the hardware machine, or both.

Core Functions

  1. Listen on a fixed port (commonly port 80 for HTTP, 443 for HTTPS, or custom ports like 8000, 8080).
  2. On incoming request, run some code and return a result.
  3. Send standard headers as part of the result (status code, content type, content length, etc.).
  4. Output can be text or other format — specified via MIME types.
What a web server does:
– A web server accepts incoming network connections on a fixed port. ✓
– A web server responds to client requests in specific ways. ✓

What a web server does NOT do:
– A web server does NOT maintain the state of the client (HTTP is stateless). ✗
– The OS handles ports and network connections; the web server works one level above.

Examples of Web Servers

Web Server Type
Apache Tomcat Server Java-based web server and servlet container
IIS (Internet Information Server) Microsoft's web server for Windows
Mongoose Lightweight embedded web server library
Python's http.server Built-in Python module for serving files
Misconception: "Python is a web server."
Correction: Python is NOT a web server — it's a programming language that can be used to run web server software. Apache Tomcat, IIS, and Mongoose are web servers; Python is not.

Key Takeaways

* Any computer with a network connection can act as a web server.
* A web server accepts incoming connections on a fixed port and responds in specific ways.
* The computer running the web server program is also called a "web server."
* A web server is used to store, process, and deliver web pages to clients.
* A web server is a computer or virtual machine used to run web applications — all of the above are true.

11. HTTP Protocol – Requests, Responses, Headers, Status Codes

Definition

HTTP (HyperText Transfer Protocol) is a text-based protocol that defines how clients and servers communicate. It specifies the format of requests (GET, POST, etc.) and responses (status codes, headers, data). HTTP is an abbreviation for Hypertext Transfer Protocol — it is a text-based protocol, not an image-based protocol.

Typical HTTP Request

GET / HTTP/1.1
Host: localhost:1500
User-Agent: curl/7.64.1
Accept: */*

Components: GET is the method; / is the path; HTTP/1.1 is the protocol version. Host specifies which host to connect to (required in HTTP/1.1 for virtual hosting). User-Agent identifies the client. Accept specifies what MIME types the client can handle — */* means "accept any form of data." A blank line signals the end of headers.

Typical HTTP Response

HTTP/1.1 200 OK
Server: SimpleHTTP/0.6 Python/3.9.2
Date: Thu, 17 Jun 2021 03:11:15 GMT
Content-type: text/html
Content-Length: 31
Last-Modified: Thu, 17 Jun 2021 03:10:18 GMT

<h1>Hello world</h1>

GET vs. POST

Aspect GET POST
Purpose Generally used to request a resource from the server Generally used to submit forms
Data visibility Data is visible in the URL Data is NOT visible in the URL
Use case Simple requests, search queries Complex form data, large text blocks, file uploads

Both GET and POST are methods used by HTTP to transfer data between a browser and a server.

Key HTTP Concepts

HTTP is a protocol — rules agreed to by both client and server.
HTTP is a text-based protocol — NOT image-based.
HTTP is an abbreviation for Hypertext Transfer Protocol.
FTP stands for File Transfer Protocol — NOT "File Traffic Protocol."
HTTP is stateless — it does NOT maintain client state between requests.
Both GET and POST are used by HTTP to transfer data between browser and server.
HTTP and FTP do NOT use the same port number — HTTP typically uses port 80, FTP uses ports 20/21.

MIME & CGI

MIME: Multipurpose Internet Mail Extensions — originally for email attachments, now used to specify content types (text/html, application/pdf, image/png).

CGI: Common Gateway Interface — an interface specification that enables web servers to execute an application program. It is an abbreviation for Common Gateway Interface. CGI is NOT related to transfer of data from client to server per se — it's about server-side execution.

Misconception: "MIME stands for Multimedia Intranet Mail Extension."
Correction: MIME stands for Multipurpose Internet Mail Extensions — none of the alternate options given are the correct expansion.

Misconception: "Accept: image/* means the client can understand any type of data."
Correction: Accept: image/* means the client is able to understand images only (any image format). It does NOT mean any type of data.

Misconception: "Accept: */* means the client accepts only images."
Correction: Accept: */* means the client is willing to accept any form of data.

12. Building a Simple Web Server – One-Line Shell Script

Definition

A minimal web server can be written in a single line of shell script. The simplest possible server listens on a port, and whenever a request arrives, it responds with a fixed HTTP header and some dynamic content (like the current date).

The One-Line Web Server

while true; do echo -e "HTTP/1.1 200 OK\n\n $(date)" | nc -l localhost 1500; done

Component Breakdown

Command Purpose
while true; do ... done Infinite loop — keeps the server running.
echo -e "HTTP/1.1 200 OK\n\n $(date)" Prints the HTTP response header (200 OK), two newlines (separating headers from body), and the current date as the body.
| (pipe) Redirects the output of echo into the next command.
nc -l localhost 1500 Netcat listens on localhost port 1500; when a connection comes in, it sends the piped data and exits, then the while loop restarts it.

Testing with curl

curl -v http://localhost:1500

The -v (verbose) flag shows the full request and response headers, which is very useful for debugging.

Python's Built-in HTTP Server

python -m http.server

This serves files from the current directory on port 8000 by default. It understands basic HTTP requests and provides proper headers including Content-type, Content-Length, and Last-Modified. By convention, if a directory is requested, it looks for an index.html file to serve.

URL Components

Example URL: http://myserver.com/index.html?uid=10&sid=4032013
– Protocol: http
– Host-name: myserver.com
– File-path: /index.html
– Query-string: uid=10&sid=4032013

Key Definitions

Loopback addresses: 127.0.0.1 (IPv4) and ::1 (IPv6) — addresses that refer to the local machine itself. 128.0.0.8 and 1::.1 are NOT valid loopback addresses.

0.0.0.0 in IPv6: Represented as :: or ::0 — NOT 10::1 or 01::0.

When you send a request to https://www.google.com/: From the server's point of view, the Web Browser is the client — NOT the Google home page or the Google server.

13. Web Browsers – Definition and Core Truths

Definition

A web browser is an application program used to display web documents. It sends HTTP requests to servers and receives HTTP responses, then renders the content for the user.

True statements about web browsers:
✓ A web browser is an application program to display web documents.
✓ A web browser sends an HTTP request and receives an HTTP response.

False statements:
✗ A web browser is installed in a web server machine and hosts websites. (That's a web server's job.)
✗ A web browser is a web app. (A browser is a desktop/mobile application that renders web apps.)

Key Takeaways

* The browser is the client in the client-server model for web communication.
* Browsers interpret HTML, CSS, and JavaScript to render web pages.
* From the server's perspective, the Web Browser is the client requesting data.

14. Validating IPv4 and IPv6 Addresses

IPv4 Address Rules

An IPv4 address consists of four octets (4 bytes = 32 bits), each separated by a dot. Each octet must be a decimal number between 0 and 255 (inclusive).

Valid IPv4 addresses: 192.168.64.34 ✓, 34.39.43.202 ✓
Invalid IPv4 addresses: 192.168.256.1 ✗ (256 > 255), 34.239.314.206 ✗ (314 > 255)

IPv6 Loopback and Zero Address

Loopback addresses: 127.0.0.1 (IPv4) and ::1 (IPv6) are loopback addresses. 128.0.0.8 and 1::.1 are NOT valid loopback addresses.

0.0.0.0 in IPv6: The correct representations are :: and ::0. 10::1 and 01::0 are NOT correct representations of 0.0.0.0 in IPv6.

Key Takeaways

* IPv4 octets are strictly 0–255 (8 bits each).
* Loopback addresses: 127.0.0.1 (IPv4) and ::1 (IPv6).
* 0.0.0.0 in IPv6 = :: or ::0.
* HTTPS is a protocol; the full address (https://onlinedegree.iitm.ac.in/academics.html) is a URL.

15. Web Request Lifecycle – DNS Lookup to Display

The Correct Order of Tasks

When you request a URL like http://myserver.com/index.html, the following tasks occur in this exact order:

  1. (4) DNS Lookup: The web browser connects to the DNS server to get the server IP address for myserver.com.
  2. (1) HTTP Request: The web browser sends an HTTP request to the server, requesting a copy of index.html.
  3. (3) Server Response: The server responds either with the requested resource or an error code.
  4. (2) Browser Display: The web browser assembles the response and displays it.

Correct sequence: 4 → 1 → 3 → 2

Misconception: "The browser sends the HTTP request before looking up the IP address."
Correction: DNS resolution must happen first — the browser cannot send an HTTP request to a server without knowing its IP address.

Key Takeaways

* DNS resolution (domain → IP) always precedes the HTTP request.
* The browser assembles and displays the response only after receiving it from the server.
* The sequence is: DNS → HTTP Request → Server Response → Browser Display.

16. Website Performance – Latency, Bandwidth, Memory, and Storage

Definition

Website performance is determined by several factors: latency (round-trip time for requests), bandwidth (data transfer rate), server memory (RAM), and storage capacity. These constraints dictate how many concurrent users a server can handle and how quickly it can respond.

1. Latency (Round Trip Time)

RTT stands for Round Trip Time — NOT "Round Tip Time," "Robust Travel Time," or "Round Trip Travel."

Speed of light in fiber cable ≈ \(2 \times 10^8\) m/s ≈ 5 ns/m.

For a distance of 1,500 km: one-way time = \(\frac{1,\!500,\!000}{2 \times 10^8} = 7.5\) ms. RTT = 15 ms.

For a data center 12,500 km away: RTT = \(\frac{2 \times 12,\!500,\!000}{2 \times 10^8} = 0.125\) s = 125 ms.

As the distance between client and server increases, the maximum number of requests per second decreases.

2. Bandwidth Calculation

Bandwidth needed = requests/sec × response size (bytes) × 8 (bits per byte).

Example: 6,250 requests/sec × 200 kB each = 6,250 × 200 × 8 = 10,000,000 kb/s = 10 Gbps.
Example 2: 10,000 requests/sec × 150 kB = 10,000 × 150 × 8 = 12,000,000 kb/s = 12 Gbps.

3. Memory (RAM) Calculation

One Python HTTP server process ≈ 6 MB. For 5 million concurrent viewers of a 2 MB video: 5,000,000 × 2 MB = 10,000,000 MB = 10 TB of RAM minimum.

4. Storage

Google indexes hundreds of billions of web pages. Estimated total index size: ~100 Petabytes (100,000,000 GB).

Key Takeaways

* RTT = Round Trip Time — NOT "Round Tip Time."
* Latency is fundamentally limited by the speed of light.
* As distance increases, max requests per second decreases.
* Scaling requires distributing load across many servers.
* Bandwidth (bps) = requests/sec × response size (bytes) × 8.
